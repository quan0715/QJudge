## **1. 工作目標（Goal）**

1. 全面檢視並更新專案的文件內容，確保：
   - 所有描述皆與目前程式碼、設定、CI/CD 流程一致。
   - 不再保留明顯過期或錯誤的資訊。
2. 依照專案現況，補齊下列文件類型（若已存在則更新）：
   1. 系統功能介紹
   2. 如何使用
   3. API 與 Model
   4. Deploy
   5. Testing
3. 若現階段功能尚未實作，**不得假裝已完成**，只能：
   - 清楚標註為「未來規劃 / Roadmap」
   - 或不寫入主流程，只在 Roadmap / TODO 中說明

---

## **2. 分支與檔案操作規範**

1. **只能在 \*\***documentation\***\* 分支上作業**
   - 任何修改、建立、刪除檔案，都必須發生在 **documentation** 分支。
2. **只允許修改文件相關檔案** **：**

- 允許修改：
  - README.md
  - **docs/** 目錄底下的 Markdown / 文檔檔案（例如：**docs/\*.md**、**docs/\*.mdx**）
  - **其他明顯屬於文件用途的檔案（例如：**CONTRIBUTING.md**, **API.md**, **DEPLOY.md**, **TESTING.md** 等）**
- 不允許修改：
  - src/**、**app/**、**lib/**、**backend/** 等程式碼目錄**
  - 測試程式、設定檔、CI 設定、Dockerfile、專案配置檔等
- 如果需要新增或修改程式碼才能讓文件變「更正確」，**也不能由你來動程式碼**，只能：
  - 在文件中標註「目前行為為 X，未來建議調整為 Y」
  - 或在 Roadmap 區塊中提出建議

1. 如需新增文件檔案，優先採用 Markdown（**\*.md**），並放在 **docs/** 目錄或專案既有慣例位置。

---

## **3. 文件架構與內容規劃**

你需要確保以下 5 類文件都有清楚的入口與內容（可以是單一文件內多個章節，也可以拆成多個檔案）：

### **3.1 系統功能介紹（System Overview / Features）**

**目的：** 讓新讀者在幾分鐘內理解這個專案「是什麼」、「能做什麼」。

內容建議包含：

- 專案簡介（What / Why）
- 核心功能列表與簡短說明
- 主要使用情境（Use Cases）
- 系統整體架構概觀（可用簡單文字描述，若專案已有架構圖可在文件中引用）
- 若有多個子服務 / 模組，簡單說明各自職責與關係

寫作原則：

- 以「功能與用途」為主，避免行銷式語氣
- 描述要與目前實作一致，不要先寫「計畫中」的功能
- 未來想做的東西集中放到「Roadmap / Future Work」章節

---

### **3.2 如何使用（Getting Started / Usage）**

**目的：** 讓開發者與使用者可以順利把專案跑起來、完成基本操作。

內容建議包含：

1. 安裝與前置需求
   - 環境需求（Node/ Python / Java / DB 版本等）
   - 安裝步驟（clone repo、安裝套件等等）
2. 本地啟動專案步驟
   - **需要執行的指令（例如 **npm install**, **npm run dev**）**
   - 若有 Docker / docker-compose，可說明如何啟動
3. 基本使用流程
   - 最簡單的「第一個成功案例」（Hello World / Demo Flow）
   - 重要 CLI 指令或 UI 操作說明
4. 常見問題（FAQ）
   - 例如常見錯誤訊息、權限問題、環境變數設定

寫作原則：

- 所有指令必須是實際可執行的（存在於 repo 的 script / make target）
- 若不確定某指令目前是否有效，就不要加入主流程，可改在「疑似過期 / 待確認」區塊註記
- 例子要盡量「最小化」：用最少步驟完成一個明確結果

---

### **3.3 API 與 Model（API & Data Models）**

**目的：** 讓前後端開發者、整合者可以正確呼叫 API、理解資料結構。

內容建議包含：

1. API Overview
   - 說明是否為 REST / GraphQL / gRPC 等
   - 說明 API Base URL 或 endpoint 結構
2. 主要 API 列表
   - 每個 API 至少包含：
     - Method + Path（例如：**GET /api/problems**）
     - 功能描述
     - Request 參數（Path / Query / Body）
     - Response 範例（含欄位說明）
     - 可能的錯誤狀態碼與含義（若有）
3. Data Models
   - 對外公開的資料結構（例如：Problem, User, Submission…）
   - 每個欄位的型別與意義
   - 若有 enum / 狀態碼，列出可能值與說明

寫作原則：

- **只能紀錄目前實作中存在的 API 與欄位** **：**
- 若 code 中沒有該 endpoint，就不能在文件中當作已實作 API
- 不確定的功能可以放在「API Roadmap / 設計草案」章節
- 範例 JSON 必須是語法正確的 JSON，並與當前 Model 結構一致
- 盡量使用表格整理欄位（欄位名稱 / 型別 / 是否必填 / 說明）

---

### **3.4 Deploy（Deployment）**

**目的：** 讓維運或開發者可以依文件成功部署專案到指定環境（例如：Production / Staging）。

內容建議包含：

1. 支援的部署方式
   - 例如：Docker、Kubernetes、傳統 VM、PaaS（Render, Heroku…）等
   - 若目前只有「本地開發環境」，就在文件中說明「尚未提供正式部署範例」，並放在未來規劃
2. 目前實際存在的部署流程
   - 若 repo 中有 **docker-compose.yml**、**k8s/** manifest、**deploy/** 腳本，說明如何使用
   - 說明必要的環境變數與密鑰（不用放實際值，只說明名稱與用途）
3. CI/CD 簡述（若有）
   - 例如 GitHub Actions / GitLab CI 如何 build / test / deploy
   - 僅描述目前 pipeline 中「已實作」的步驟

寫作原則：

- 不要虛構部署能力：若現在沒有 k8s manifest，就不要聲稱「支援一鍵部署到 Kubernetes」
- 對於尚未整理好的部署方式，可以在「Future Work」中寫出「規劃中的部署目標」
- 指令與檔案路徑要與 repo 實際內容一致

---

### **3.5 Testing（測試）**

**目的：** 讓貢獻者與維護者知道如何執行測試、了解測試策略。

內容建議包含：

1. 測試種類說明
   - 例如：Unit tests, Integration tests, E2E tests
   - 目前專案實際有的種類才寫
2. 如何執行測試
   - **實際指令：如 **npm test**, **pytest**, **go test ./...** 等**
   - 若有不同 test suite，說明各自用途與指令
3. 測試覆蓋率（若有）
   - 如何產生 coverage report
4. 測試相關的約定
   - 例如：檔名規則（**\*.test.ts**）、目錄結構（**tests/**）、mock 策略簡述

寫作原則：

- 不要宣稱「測試覆蓋率很高」或「有 E2E 測試」，除非 repo 中真的有對應測試與工具
- 若目前測試非常少，也要誠實描述現況，可以加上「未來測試擴充計畫」章節

---

## **4. 寫作風格與格式**

1. 文件格式：
   - 優先使用 Markdown（**\*.md**）
   - 標題層級合理使用：**#**（文件標題）、**##**（大章）、**###**（小節）
2. 語氣與風格：
   - 清楚、專業、中性，避免過度誇飾或行銷語言
   - 以工程師與貢獻者為主要讀者
3. 語言：
   - 預設沿用專案現有主要語言：
     - 若 README 大多為英文 → 新文件也以英文撰寫
     - 若專案明顯以繁體中文為主 → 優先使用繁體中文，技術名詞保留英文
4. 內容要求：
   - 僅描述「已實作」的功能與狀態
   - 對於未來想做的事情，統一放在「Roadmap / Future Work」區塊，用條列方式描述計畫
5. 範例與程式碼區塊：
   - **使用正確的程式語言標記：如** `bash**、** `ts**、** ```json
   - 範例指令要簡潔，盡量能直接複製貼上執行
   - JSON 範例必須是合法語法，不要出現尾逗號

---

## **5. 嚴格禁止事項（Forbidden）**

在任何情況下，你都必須遵守以下限制：

1. **不可修改任何非文件檔案**
   - 不得修改程式碼、設定檔、CI/CD 流程、Dockerfile、建構腳本等
   - 不得新增或刪除與文件無關的檔案
2. **不得在文件中聲稱「尚未實作」的功能已存在**
   - 不可以自己想像未來功能、API、部署方式，當作目前已支援
   - 如果功能還沒實作：
     - 可在「Future Work / Roadmap」或「設計草案」章節說明
     - 必須清楚標註為「規劃中」、「尚未實作」，避免誤導使用者
3. 若你無法百分之百確定目前實作是否支援某功能：
   - 寧願保守不寫，或用「待確認 / TODO」標註
   - 不得假設「應該有」就直接寫入主文件內容

---

## **6. 輸出要求**

當你被要求更新或撰寫文件時：

1. 只產生**文件內容本身**（Markdown / 文字內容），不要輸出 git 指令或 commit 訊息，除非使用者特別要求。
2. 若是新增文件，請提供：
   - **檔名建議（例如：**docs/overview.md**, **docs/deploy.md**）**
   - 完整檔案內容
3. 若是修改既有文件，請輸出：
   - 更新後的完整段落 / 完整檔案內容（方便直接覆蓋）
   - 避免只給「diff 片段」讓人難以操作
