# Design: 考試、作業、練習 — 場景語意與最小資料契約

**日期：** 2026-04-05  
**狀態：** 規格草稿（供審閱；尚未進入實作計畫）  
**範圍：** 在既有 **Contest（競賽）** 模組上，以**最小改動**區分三種課堂常見行為，不討論 UI 細節與完整遷移腳本。

---

## 1. 背景與現況

後端 `Contest` 已具備：

- `delivery_mode`: `exam`（正式考試流程）與 `practice`（說明文字涵蓋「教室練習/作業流程」）。
- 與時間、可見性、防作弊、題目鎖定等既有欄位。

本規格將 **「作業」與「練習」** 從同一 `practice` 軌道中，用**少量語意欄位與行為規則**拆開使用者心智，避免大拆模組。

---

## 2. 三種場景的產品定義

### 2.1 考試（Exam）

- **風險與契約**：高；通常同步時間窗、少次或單次提交、成績敏感。
- **對應**：`delivery_mode = exam`（維持現有「正式考試流程」敘事與相關規則）。
- **資料**：提交與批改紀錄需**完整可追溯**，供爭議處理與成績採計。

### 2.2 作業（Homework，偏 take-home exam）

- **風險與契約**：中～高；學生帶回家完成，**結果會保存且具「要交差／要評分」的語意**。
- **對應**：`delivery_mode = practice`（或日後若改名為 `assignment`，語意同等），且 `**counts_toward_grade = true`**（名稱可為等價布林／枚舉，下稱「**計分契約**」）。
- **時間**：以截止日、允許遲交等政策為主（具體旗標沿用或擴充現有欄位，本稿不鎖定欄位名）。
- **資料**：需保留足夠歷史與版本，供老師批改與成績採計（與現行作業流一致方向）。

### 2.3 練習（Practice）

- **風險與契約**：低；**不算分**（不進正式成績簿語意），鼓勵反覆操作。
- **對應**：`delivery_mode = practice` 且 `**counts_toward_grade = false`**。
- **進入與提交**：**不限制**重新進入與重送次數（無限期次嘗試）。
- **資料 — 儲存策略（已決）**：採 **「只保留最近一次有效提交」**（覆寫語意），以降低儲存與產品複雜度；不維護練習用多版本歷史。若日後需要學習分析歷史，以**獨立專題**擴充，不破壞本契約。

---

## 3. 行為對照（摘要）


| 場景  | `delivery_mode` | 計分契約    | 嘗試／重送 | 提交儲存策略（原則） |
| --- | --------------- | ------- | ----- | ---------- |
| 考試  | `exam`          | 是（正式成績） | 依既有設定 | 完整稽核       |
| 作業  | `practice`      | 是       | 依作業政策 | 需可追溯／供採計   |
| 練習  | `practice`      | 否       | 不限    | 僅保留最新一次    |


---

## 4. 設計原則

1. **不重複建「作業實體」**：仍以 **Contest 實例** 表達一場作業或練習，透過 **計分契約 + 儲存策略** 區分。
2. **語意優先於字面**：若 UI 顯示「作業／練習／考試」，由 `delivery_mode` 與計分契約**衍生**，避免三個獨立模組分岔。
3. **練習採最新一筆**：實作時須定義「覆寫」的**粒度**（例如每題、每卷、或每場 contest）；具體粒度在 **implementation plan** 階段依既有 submission 模型細訂。
