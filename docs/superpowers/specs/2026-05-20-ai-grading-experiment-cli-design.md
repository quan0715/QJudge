# AI Grading Experiment CLI Design

Date: 2026-05-20

Note: QJudge Paper CLI V1 is now defined separately as an OAuth-first
product-like CLI frontend in
`docs/superpowers/specs/2026-05-20-qjudge-paper-cli-oauth-first-design.md`.
This document remains the experiment design for multi-model grading,
cross-review, and aggregation after the V1 product flow is stable.

## Purpose

This design defines a low-engineering-cost experiment workflow for evaluating AI-assisted grading inside QJudge. The thesis scope remains the end-to-end online exam system: exam setup, student answering, anti-cheat monitoring, submission, grading, review, and result publication. AI grading is the final output of that system and needs experimental validation for accuracy, reliability, cost, and review workload reduction.

The first implementation target is a CLI workflow, not a frontend product flow. The CLI lets us run controlled experiments such as:

```bash
./scripts/experiments/run-test-exam exam1 --question-id <question_id>
./scripts/experiments/run-test-exam exam2 --question-id <question_id>
./scripts/experiments/run-test-exam exam2 --full
```

The CLI should reuse the existing `ai-service` runtime, DeepAgent runner, model factory, MCP grading tools, and AI artifact storage. It should not rewrite the current frontend AI grading screen.

## Research Framing

The experiment has two layers.

| Layer | Purpose | Scope |
| --- | --- | --- |
| Single-question pilot | Validate prompts, per-model rubrics, grade artifacts, cross-review, and summary format | One selected question from Exam1 or Exam2 |
| Full-exam experiment | Measure total grading time, cost, score distribution, rank stability, and human review workload | All subjective questions in Exam1 or Exam2 |

The single-question pilot is required before the full-exam experiment. It reduces risk because the same artifact and review format can be verified on a small sample before scaling to all questions.

## Experiment Design

### Stage 0: Input Snapshot

For each experiment run, the system captures a stable input snapshot:

- contest id
- question id or full-exam scope
- question prompt
- max score
- reference answer / correct answer / teacher-provided answer key
- all student answers for the selected scope
- existing human score and feedback, if available

The snapshot must be recorded in `manifest.json` so repeated runs are traceable.

### Stage 1: Independent AI Grading

Each AI model works independently. The models do not see existing human scores, human feedback, or other model outputs during blind grading.

Recommended initial model set:

| Candidate | Role |
| --- | --- |
| OpenAI large model | AI grader A |
| DeepSeek thinking / large-context model | AI grader B |
| Claude large model | AI grader C |
| Human H0 | Existing human grading result, treated as the fourth candidate |

The exact provider model names are configuration decisions. In this document,
`openai-large` and `claude-large` are CLI aliases that should be mapped to
real provider model strings in the experiment config before execution.

Each AI model produces its own rubric and grade file per question:

```text
model_a/q1/rubric.md
model_a/q1/grade.csv
model_b/q1/rubric.md
model_b/q1/grade.csv
model_c/q1/rubric.md
model_c/q1/grade.csv
human_h0/q1/grade.csv
```

This is intentionally an end-to-end grading comparison. The experiment compares each model's ability to generate a usable rubric and grade answers from only the question, max score, reference answer, and student responses.

The thesis should describe this as "end-to-end AI grading ability", not "same-rubric model scoring ability".

### Stage 2: Cross Review

After blind grading, each AI model reviews the other candidates' grading results. The reviewer receives a structured review packet generated by the CLI. The packet must contain:

- question prompt
- max score
- reference answer
- student answer
- candidate score
- candidate feedback
- candidate rubric reference

The reviewer does not discover files by itself. The CLI creates explicit `review_packet.json` files.

Example output:

```text
reviews/q1/review_packet_model_a.json
reviews/q1/review_model_a.csv
reviews/q1/review_packet_model_b.json
reviews/q1/review_model_b.csv
reviews/q1/review_packet_model_c.json
reviews/q1/review_model_c.csv
```

`review.csv` uses a fixed schema:

```csv
exam_answer_id,reviewed_candidate_id,accepted,suggested_score,severity,disagreement_type,confidence,reason
```

Allowed `severity` values:

```text
none
minor
major
```

Allowed `disagreement_type` values:

```text
none
rubric_too_strict
rubric_too_loose
missing_key_criterion
wrong_interpretation
score_not_supported
feedback_not_supported
unclear_answer
unclear_rubric
```

### Stage 3: Aggregation

The CLI merges the candidate grades and review outputs into:

```text
summary.csv
report.md
```

`summary.csv` should include at least:

```csv
exam_answer_id,question_id,student_anonymous_id,max_score,human_h0_score,model_a_score,model_b_score,model_c_score,score_range,score_stddev,model_a_mae_to_h0,model_b_mae_to_h0,model_c_mae_to_h0,review_acceptance_rate,major_issue_count,rubric_disagreement_count,audit_priority_score
```

For full-exam runs, the summary also includes participant-level totals:

```csv
student_anonymous_id,human_h0_total,model_a_total,model_b_total,model_c_total,consensus_total,total_delta_vs_h0,rank_h0,rank_consensus,rank_delta
```

### Stage 4: Human Final Audit

Human audit is not part of the first CLI MVP, but the data format should support it. The CLI produces an `audit_priority_score` so the teacher can focus on high-risk answers:

- large score range among candidates
- low review acceptance rate
- major issue flagged by reviewers
- rubric disagreement
- high max-score question
- large difference from Human H0

Later, the final product flow can show these high-priority answers in the grading UI and let teachers confirm final scores.

## Metrics

### Single-Question Pilot Metrics

| Metric | Purpose |
| --- | --- |
| AI vs Human H0 MAE | Distance between each AI grader and existing human score |
| Score range / standard deviation | Degree of model disagreement |
| Review acceptance rate | How often reviewers accept another candidate's grade |
| Major issue rate | How often reviewers find serious grading problems |
| Rubric disagreement rate | How often disagreement is about the rubric rather than only score |
| Audit priority top N | Which answers should be manually checked first |
| Runtime / token / cost | Feasibility for full-exam scaling |

### Full-Exam Metrics

| Metric | Purpose |
| --- | --- |
| Total grading time | Whether one-click exam grading is operationally viable |
| Token and provider cost | Cost of large-model grading at exam scale |
| Total score distribution | Whether AI grading is globally lenient or strict |
| Rank correlation | Whether AI consensus preserves student ranking |
| Per-question MAE | Which questions are hardest for AI grading |
| High-disagreement answer rate | How much human review can be narrowed |
| Human review reduction estimate | Percentage of answers that can be skipped if only high-risk answers are audited |

## Manifest

`manifest.json` is the source of truth for artifact traceability. It records:

- experiment run id
- exam key
- contest id
- scope type: `single_question` or `full_exam`
- question ids
- model ids
- session ids
- AI artifact ids
- local artifact paths
- prompt version
- created timestamp
- whether official scores were modified

Official scores must not be modified by default.

The following JSON is an example. UUID values are placeholders for the actual
contest and question ids from Exam1 or Exam2.

```json
{
  "run_id": "exam2-20260520-153000",
  "exam_key": "exam2",
  "contest_id": "00000000-0000-0000-0000-000000000000",
  "scope": {
    "type": "single_question",
    "question_id": "11111111-1111-1111-1111-111111111111"
  },
  "write_back_scores": false,
  "models": [
    {
      "candidate_id": "model_a",
      "model_id": "openai-large",
      "provider": "openai"

      "candidate_id": "model_b",

    {
      "candidate_id": "model_c",
```

## CLI Configuration

Config files live under:

```text
experiments/grading/exam1.yml
experiments/grading/exam2.yml
```

Example:

```yaml
exam_key: exam2
contest_id: "00000000-0000-0000-0000-000000000000"

default_scope:
  type: single_question
  question_id: "11111111-1111-1111-1111-111111111111"

models:
  - candidate_id: model_a
    model_id: openai-large
    provider: openai
  - candidate_id: model_b
    model_id: deepseek-v4-thinking
    provider: deepseek
  - candidate_id: model_c
    model_id: claude-large
    provider: anthropic

stages:
  blind_grading: true
  cross_review: true
  aggregate: true

write_back_scores: false
output_dir: reports/ai-grading
```

## Engineering Plan

### Phase 1: CLI Skeleton

Add:

```text
ai-service/experiments/grading/cli.py
ai-service/experiments/grading/runner.py
ai-service/experiments/grading/prompts.py
ai-service/experiments/grading/manifest.py
ai-service/experiments/grading/artifact_downloader.py
scripts/experiments/run-test-exam
```

The CLI loads exam config, resolves scope, creates a run id, creates or reuses backend `AISession` rows, then calls `DeepAgentRunner.run_stream()`.

### Phase 2: Single-Question Blind Grading

For each configured AI model:

1. create a deterministic session for `run_id + model + question`
2. seed `grade.csv` with student answers
3. prompt the model to produce `rubric.md`
4. prompt the model to fill `score` and `reason` in `grade.csv`
5. download artifacts locally
6. update `manifest.json`

Human H0 is exported from existing `ExamAnswer.score` and `ExamAnswer.feedback` into `human_h0/q1/grade.csv`.

### Phase 3: Claude Provider

Current `ai-service` supports OpenAI and DeepSeek. The experiment needs Claude as the third large model.

Add:

- `langchain-anthropic`
- `ANTHROPIC_API_KEY`
- `claude-large` canonical model id
- Anthropic branch in `ModelFactory`
- model contract tests

For MVP, Claude can be CLI-supported before it is exposed in the frontend model picker.

### Phase 4: Cross Review

Generate review packets from the manifest and candidate grade files. Each AI model reviews other candidates and writes `review.csv`.

Reviewer prompts should require structured output only. The reviewer should cite candidate ids and disagreement types rather than producing free-form prose only.

### Phase 5: Summary and Report

Generate:

- `summary.csv`
- `report.md`

The report should include:

- model list
- question count
- answer count
- runtime
- cost estimate when token usage is available
- per-model MAE vs Human H0
- disagreement summary
- top audit-priority answers

### Phase 6: Full-Exam Mode

Extend the same pipeline from one question to all subjective questions.

Full-exam mode loops over question ids, then adds participant-level totals and rank comparisons.

## Non-Goals

The CLI MVP does not:

- write AI grades back to official `ExamAnswer.score`
- replace the current frontend AI grading screen
- implement the final teacher audit UI
- require teachers to handwrite rubrics
- require all experiments to use a shared canonical rubric

## Safety and Data Integrity

Default behavior:

```text
write_back_scores: false
```

The experiment must never modify official grades unless an explicit future `--apply-final` mode is implemented.

Student identifiers in exported CSV should use anonymized ids when used for thesis analysis.

## Estimated Engineering Effort

| Work item | Estimate |
| --- | ---: |
| CLI skeleton and config loading | 0.5-1 day |
| Single-question grading orchestration | 2-3 days |
| Artifact download and manifest | 1 day |
| Claude provider support | 1 day |
| Human H0 export | 0.5 day |
| Summary metrics | 1 day |
| Cross-review packets and review runs | 2-3 days |
| Full-exam mode | 2-3 days |

Single-question MVP: 4-6 working days.

Single-question plus cross-review: 7-10 working days.

Full-exam experiment: additional 2-3 working days after the single-question path is stable.

## Open Decisions

1. Which exact OpenAI large model should be used for `model_a`?
2. Which exact Claude model should be used for `model_c`?
3. Which Exam1 and Exam2 question ids should be used for the pilot?
4. Should full-exam mode include only subjective questions, or all non-objective questions with written explanations?
5. Should Human H0 be treated as a reviewer candidate in cross-review, or only as a comparison baseline?

Recommended defaults:

- use one pilot question from Exam2 first
- include Human H0 as a candidate in review packets
- keep official scores read-only
- treat the experiment as end-to-end grading, including model-generated rubrics
